# S3 Storage, Scheduling, Retention, and HA Reseed Plan

## Goal

Add first-class off-box backup storage and operations on top of the per-database
logical backup format:

- write backups directly to S3-compatible storage
- schedule FULL, DIFFERENTIAL, and LOG backups for one or more databases
- automatically react to primary changes/failover
- search available backup ranges
- restore PITR by timestamp without manually listing every `.bak`
- safely prune old backups without breaking restore chains

S3 support must not weaken the existing guarantee: a completed LOG/DIFF backup
is only considered durable after its `.bak` and manifest are safely written to
remote storage and verified.

## Decisions

These are the v1 design decisions for this feature.

| Area | Decision |
|------|----------|
| Credentials | Read S3 credentials from environment variables. Do not store raw S3 secrets in catalog tables, backup files, or manifests. |
| S3 client | Implement a native S3 client in the extension. Use low-level dependencies such as OpenSSL and libcurl where appropriate, but own S3 behavior: multipart state, retries, verification, manifests, Object Lock, and catalog integration. |
| Scheduler | Run schedules from PostgreSQL background workers. `shared_preload_libraries = 'pg_dbbackup'` is already required for FULL mode. |
| Multipart cleanup | Track multipart uploads in the catalog and abort stale uploads from workers. Also recommend/enforce an S3 lifecycle rule for incomplete multipart cleanup. |
| Catalog rebuild | Search and restore perform automatic lazy import from S3 manifests when the local catalog is empty or stale. |
| Object Lock | Support S3 Object Lock where the provider supports it. |
| Time semantics | PITR/search APIs use `timestamptz`; schedules use named time zones. Store artifact times as UTC. |
| Streaming | Implement streaming zstd/AES framing before S3 upload support; do not require full-file local staging for normal operation. |
| Cost controls | Include concurrency limits, multipart part sizing, retry/backoff, bandwidth throttle, size estimates, storage usage, and budget warnings. |
| Restore drills | Manual restore-drill jobs only; do not schedule automatic drills by default. |
| Alerts | Use event tables and `NOTIFY` as source of truth, plus optional async webhooks. |
| Failover membership | Snapshot backup-set membership when a primary-change event starts. |
| Restore mode | Hybrid restore: stream from S3 by default, with optional local staging path. |
| Remote catalog authority | Per-artifact manifests are authoritative. `index.jsonl` and local catalog rows are rebuildable caches. |
| External lifecycle | Detect externally deleted objects and mark affected ranges unavailable. |
| API names | Behavior is fixed by this plan, but SQL function names/signatures can change during implementation before v1 freeze. |
| Slot advancement | Advance logical slots only after durable backup coverage exists for the advanced LSN range. |

## Storage Targets

Storage targets are named destinations. A target can be local filesystem or
S3-compatible object storage.

Supported target types:

- `local`: server-side path prefix
- `s3`: AWS S3
- `s3-compatible`: MinIO, Cloudflare R2, Backblaze B2, Wasabi, etc.

Example API:

```sql
SELECT dbbackup.create_s3_target(
  name := 'prod_s3',
  bucket := 'db-backups',
  prefix := 'postgres/prod',
  region := 'eu-west-1',
  endpoint_url := NULL,
  force_path_style := false,
  encryption := 'sse-kms',
  kms_key_id := 'alias/postgres-backups'
);
```

Credentials are read from environment variables in v1. Standard variables:

- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- `AWS_SESSION_TOKEN`
- `AWS_REGION`
- optional endpoint-related variables when useful for S3-compatible storage

Storage targets store only non-secret configuration: bucket, prefix, region,
endpoint URL, path-style flag, and encryption/object-lock settings. `.bak`
password encryption remains independent from S3 server-side encryption. S3
credentials must never be written into backup headers or manifests.

IAM/workload identity can be added later without changing backup metadata.

## Native S3 Client

The extension owns S3 behavior. It should use low-level dependencies where
reasonable, but not rely on external command-line tools.

Required client features:

- SigV4 request signing
- `PUT`, `GET`, `HEAD`, `LIST`
- multipart create/upload/complete/abort
- S3-compatible endpoints
- path-style and virtual-hosted-style addressing
- retry/backoff with retryable error classification
- object size and checksum verification
- request timeout controls
- bandwidth throttling
- Object Lock headers when configured

The native client avoids requiring `aws`, `rclone`, or other external tools on
every PostgreSQL server.

Client scan result:

- `aws-c-s3` is the official AWS C99 S3 client and is high-performance, but it
  brings the AWS CRT dependency stack. That is too heavy for this extension's
  default packaging target.
- `libs3` is a small C S3 library and has multipart APIs, but it is old,
  LGPL/GPL licensed, has no release stream, and would still need careful review
  for modern S3 features such as Object Lock, modern checksum headers, and
  provider compatibility.
- `libcurl` has AWS SigV4 support and is already a realistic PostgreSQL
  extension dependency on many platforms. The preferred v1 path is a thin
  pg_dbbackup-owned S3 layer using libcurl/OpenSSL for HTTP/signing primitives,
  while pg_dbbackup owns multipart state, manifests, retries, checksums,
  Object Lock semantics, and catalog integration.

If libcurl's SigV4 support is unavailable on a target platform/version, provide
a small internal SigV4 signer rather than pulling in a larger SDK.

## Backup Sets

A backup set is an operations grouping. Each database still has its own chain,
logical slot, artifacts, and restore range.

Example:

```sql
SELECT dbbackup.create_backup_set(
  name := 'prod_core',
  storage_target := 'prod_s3',
  on_unsafe_failover := 'full_now'
);

SELECT dbbackup.add_database_to_backup_set('prod_core', 'app');
SELECT dbbackup.add_database_to_backup_set('prod_core', 'billing');
SELECT dbbackup.add_database_to_backup_set('prod_core', 'analytics');
```

Mutable membership:

```sql
SELECT dbbackup.remove_database_from_backup_set(
  backup_set := 'prod_core',
  dbname := 'analytics',
  close_chain := true,
  keep_history := true
);

SELECT dbbackup.set_backup_set_databases(
  backup_set := 'prod_core',
  databases := ARRAY['app', 'billing']
);

SELECT * FROM dbbackup.list_backup_set_databases('prod_core');
```

Removal rules:

- never delete S3 objects just because a database is removed from a set
- `close_chain := true` takes a final LOG if possible, then marks the chain
  closed
- `keep_history := true` preserves restore/search metadata
- physical deletion is handled only by retention

Database deletion policy:

- `pg_dbbackup` does not automatically make a final backup before
  `DROP DATABASE`
- developers/operators must explicitly run a final FULL/LOG backup before
  dropping a database if they need a final recovery point
- if a protected database disappears while still in a backup set, mark it
  `dropped`, disable future jobs for that database, keep history, and emit an
  event
- never delete S3 artifacts automatically because a PostgreSQL database was
  dropped

## Schedules

Schedules attach to backup sets, not individual databases. The scheduler expands
the set into one job per database.

Example:

```sql
SELECT dbbackup.create_schedule(
  backup_set := 'prod_core',
  name := 'nightly-full',
  backup_type := 'full',
  cron := '0 4 * * *',
  timezone := 'Europe/Amsterdam'
);

SELECT dbbackup.create_schedule(
  backup_set := 'prod_core',
  name := 'log-10m',
  backup_type := 'log',
  every := interval '10 minutes'
);

SELECT dbbackup.alter_schedule(
  backup_set := 'prod_core',
  name := 'log-10m',
  every := interval '5 minutes'
);

SELECT dbbackup.pause_schedule('prod_core', 'log-10m');
SELECT dbbackup.resume_schedule('prod_core', 'log-10m');
SELECT dbbackup.drop_schedule('prod_core', 'log-10m');
```

Schedule identity rules:

- every schedule has an immutable `schedule_id`
- every schedule also has a required human-readable `name`
- `name` is unique inside a backup set
- `create_schedule` returns the `schedule_id`
- operators can alter, pause, resume, or drop by `(backup_set, name)` or by
  `schedule_id`
- changing a schedule affects future runs only; already queued/running jobs keep
  their original job definition
- dropping a schedule does not delete backup artifacts or catalog history

Backup windows can wrap midnight:

```text
window 22:00-06:00 means local_time >= 22:00 OR local_time < 06:00
```

FULL backup scheduling rule:

- a FULL backup that starts a PITR chain creates a persistent logical slot for
  that database if one does not already exist
- a replacement FULL reuses the active database slot and advances it to the new
  FULL boundary only after the new FULL is durable
- when replacing an existing active chain, close the old chain with a tail LOG
  up to the new FULL snapshot boundary when the database is in FULL mode
- upload and verify the tail LOG first
- upload and verify the new FULL and manifest
- only after both the old tail LOG and new FULL are durable, mark the new chain
  active and mark the old chain closed

This avoids a PITR coverage gap between the last scheduled LOG and the new FULL
snapshot. If a tail LOG cannot be taken, record the old chain as incomplete and
warn before starting the new FULL.

Slot lifecycle rule:

- keep the logical slot for the currently active FULL chain
- do not drop the current chain's slot after its initial FULL if LOG backups are
  expected
- do not drop/recreate a healthy active slot just to take a replacement FULL;
  advance it after the replacement FULL is durable
- if the recorded slot is missing, stale, or belongs to a broken chain, a new
  FULL can recreate the database slot and start a new chain
- if a FULL backup is explicitly one-shot/SIMPLE with no PITR LOG chain, no
  persistent logical slot is needed after the backup completes

Slot advancement rule:

- the active chain slot moves forward whenever a backup durably covers a newer
  LSN range
- LOG backup: advance/confirm the active slot to the LOG stop LSN after the
  LOG `.bak` and manifest are durable
- DIFFERENTIAL/INCREMENTAL backup: advance/confirm the active slot to the backup
  stop LSN only if the artifact includes complete database coverage through that
  LSN, including required DDL, sequence, large-object, and extension frames
- replacement FULL backup: advance the active database slot to the new FULL
  snapshot boundary after the old tail LOG and new FULL are durable
- never advance or drop a slot for a range that exists only in memory, a temp
  file, an incomplete upload, or an unverified manifest

RPO note:

- running LOG backups only from 22:00-06:00 means daytime changes are retained
  by the logical slot but are not off-box until the next LOG upload
- for production PITR, LOG backups should usually run all day
- FULL backups can be limited to a night window

## Background Jobs and Progress

Long-running backup, upload, restore, and retention work should return a job id
immediately. Scheduled jobs are driven by PostgreSQL background workers loaded
through `shared_preload_libraries`.

Worker responsibilities:

- wake periodically
- act only when `NOT pg_is_in_recovery()`
- scan due schedules
- queue backup jobs
- detect primary/timeline changes
- run automatic failover reseed checks
- apply retention
- clean stale multipart uploads
- deliver webhook events asynchronously

Example:

```sql
SELECT dbbackup.pg_dbbackup_to_storage(
  dbname := 'app',
  type := 'log',
  storage_target := 'prod_s3'
);

SELECT * FROM dbbackup.backup_job_status('<job_id>');
SELECT * FROM dbbackup.backup_job_events('<job_id>');
```

Job fields should include:

- `job_id`
- `backup_set`
- `dbname`
- `type`
- `status`
- `phase`
- `bytes_done`
- `bytes_total`
- `current_artifact_uri`
- `error_code`
- `error_message`
- `started_at`
- `finished_at`

Manual functions should still exist for tests and operator override, but the
product behavior is automatic.

Concurrency:

- advisory lock per database and chain
- no overlapping FULL/DIFF/LOG jobs for the same database chain
- one failover reseed job per database per promotion event
- scheduler must be idempotent

## Upload Semantics

Large backups require multipart upload. Multipart uploads are catalog-tracked so
crashed jobs can be cleaned up. Buckets should also have an incomplete multipart
upload lifecycle rule as defense in depth.

Required behavior:

1. Create backup stream.
2. Upload with multipart when size crosses threshold.
3. Compute checksums while streaming.
4. Abort incomplete multipart upload on failure.
5. Verify uploaded object size/checksum where the backend supports it.
6. Upload sidecar manifest.
7. Update local catalog.
8. Only then mark backup archived and advance/confirm chain state.

LOG/DIFF safety rule:

- do not advance the chain or mark the range restorable until the remote `.bak`
  and manifest are durable

LOG cleanup/checkpoint rule:

- `pg_dbbackup` must never directly delete or truncate PostgreSQL WAL files
- after a LOG backup's `.bak` and manifest are durable, advance the chain
  checkpoint and confirm/advance the logical slot through the backed-up LSN
- PostgreSQL can then recycle WAL normally when no replication slot, standby,
  archive command, checkpoint, or other server rule still needs it
- local `pg_dbbackup` journal/spool rows, such as DDL, sequence, large-object,
  or future hook-captured frames, can be pruned only after every active chain
  that needs them has a durable LOG/DIFF artifact covering the row
- failed uploads do not advance the chain checkpoint, do not confirm the slot
  past the failed range, and do not prune local journal rows
- pruning should be batched and visible in job progress/events so a large
  journal cleanup does not surprise operators

The normal path streams directly into the multipart upload. If local staging is
explicitly requested, use a temp path and delete it after successful upload.

## Streaming `.bak` Writer

S3 support depends on a streaming writer abstraction. Do not bolt S3 onto a
whole-file staging-only path.

Required shape:

```text
backup producer
  -> bak frame writer
  -> checksum sink
  -> compression filter
  -> encryption filter
  -> local file sink OR S3 multipart sink
```

Required behavior:

- stream table `COPY` payloads directly into `.bak` sections
- stream zstd frames
- stream AES-GCM with chunk framing that can be verified on restore
- compute checksums while writing
- keep the backup header inspectable without the backup password
- support restore streaming from S3 through decrypt/decompress/read frames

## S3 Layout

Proposed object layout:

```text
s3://bucket/prefix/
  targets/prod_s3.json
  db/app/index.jsonl
  db/app/chains/<chain_id>/chain.json
  db/app/chains/<chain_id>/full_<created_at>_<backup_id>.bak
  db/app/chains/<chain_id>/full_<created_at>_<backup_id>.manifest.json
  db/app/chains/<chain_id>/diff_<start>_<stop>_<backup_id>.bak
  db/app/chains/<chain_id>/diff_<start>_<stop>_<backup_id>.manifest.json
  db/app/chains/<chain_id>/log_<start>_<stop>_<backup_id>.bak
  db/app/chains/<chain_id>/log_<start>_<stop>_<backup_id>.manifest.json
```

The `.bak` file remains the authoritative backup artifact. The sidecar manifest
is authoritative for discovery and remote catalog rebuild. `index.jsonl` is only
a cache.

Manifest contents:

- artifact URI
- backup id
- chain id
- database name and source database OID
- PostgreSQL version
- backup type
- mode
- created time
- range start time
- range end time
- start LSN
- stop LSN
- previous stop LSN
- base FULL backup id
- optional DIFF base id
- checksum
- compressed/encrypted flags
- required extensions/adapters
- file size
- upload status

The manifest must not contain row data or secrets.

## Local Catalog

Suggested tables:

```text
storage_targets
backup_sets
backup_set_databases
schedules
backup_jobs
backup_chains
backup_artifacts
backup_chain_ranges
restore_jobs
retention_runs
primary_change_events
```

The local catalog is an acceleration/index layer. It must be rebuildable from
S3 manifests after total cluster loss.

Search and restore must automatically import needed manifest metadata when the
local catalog is empty or stale. A manual refresh function can exist for
operators, but normal restore should not require it.

## Search

Users should be able to search by database and time range. Time range arguments
must be `timestamptz`.

```sql
SELECT *
FROM dbbackup.search_backups(
  dbname := 'app',
  from_time := '2026-05-01 00:00+00'::timestamptz,
  to_time := '2026-05-12 12:00+00'::timestamptz,
  storage_target := 'prod_s3'
);
```

Return chain-level coverage:

```text
chain_id
dbname
range_start
range_end
full_uri
diff_count
log_count
restorable
gap_reason
required_artifacts
```

Never report a time as restorable unless the chain is continuous and every
required remote artifact exists and verifies. If an external lifecycle policy
deleted an object, mark the artifact `missing_external`, emit a warning event,
and exclude the range from restorable coverage.

## Auto PITR Restore

Users should not need to manually assemble a file list. `stop_at` must be
`timestamptz`.

```sql
SELECT dbbackup.pg_dbrestore_at(
  dbname := 'app',
  target_db := 'app_restore',
  stop_at := '2026-05-12 03:42:10+00'::timestamptz,
  storage_target := 'prod_s3'
);
```

Restore selection algorithm:

1. Find the newest FULL before `stop_at`.
2. Find the latest useful DIFFERENTIAL after that FULL and before `stop_at`.
3. Add all LOG backups needed to cover through `stop_at`.
4. Validate LSN continuity and timestamp coverage.
5. Stream artifacts from S3 by default, or download to a configured staging path.
6. Run restore with `stop_at`.
7. Report the selected chain and artifact list in the restore job metadata.

If no continuous chain covers `stop_at`, error with nearest restorable ranges.

Restore mode:

- default: direct S3 streaming
- optional: `staging_path` for local download and full-artifact verification
- staging files are temporary and cleaned up after success/failure

## Primary Change and Automatic Reseed

This should be automatic product behavior. A manual function can exist for tests
and operator override, but users should not normally call it.

Detection options:

- scheduler polling: periodically check `pg_is_in_recovery()` and primary
  identity/timeline
- HA hook: PgDog/Patroni/Kubernetes/etc calls
  `dbbackup.pg_dbbackup_on_primary_change()`

Default failover behavior:

```text
if this node is not primary:
  do nothing

if primary identity/timeline changed:
  snapshot enabled backup sets and database membership
  for each enabled backup set:
    for each database in the event snapshot:
      check chain slot and last archived range
      if safe:
        queue immediate LOG backup to S3
      else:
        mark old chain broken
        queue immediate FULL backup to S3
        start a new chain
```

Safe means the new primary has the expected persistent failover logical slot and
can continue from the last archived chain position.

Unsafe means any of:

- expected slot is missing
- slot is temporary
- slot is invalidated
- slot is not failover-enabled
- slot belongs to the wrong DB/plugin
- slot LSN cannot continue from the last archived range

Unsafe failover policy options:

- `full_now`: default; bypass normal backup windows and take a new FULL
- `full_in_next_window`: wait for the next allowed schedule window
- `alert_only`: do not reseed automatically

Default should be `full_now`. Waiting leaves the promoted primary without fresh
off-box protection.

If the old primary is still reachable after failover, do not blindly take a tail
LOG from it unless the HA layer proves it is fenced. Avoid split-brain
assumptions.

Concurrent add/remove operations affect future primary-change events, not the
current event snapshot.

## Object Lock

Storage targets can request Object Lock when the provider supports it.

Example settings:

```text
object_lock_mode = governance | compliance | NULL
object_lock_retain_for = interval '31 days'
legal_hold = false
```

Rules:

- upload requests set Object Lock headers when configured
- target validation fails if Object Lock is requested but unsupported
- retention handles protected-object delete errors as expected protection, not
  artifact corruption
- compliance retention cannot be shortened by pg_dbbackup
- governance bypass is not used by default

## Retention

Retention is applied to chains and artifacts, not random individual files.

Sane default:

```text
PITR window: keep LOG/DIFF coverage for 14 days
FULL history: keep FULL chains for 31 days
Minimum chains: always keep newest 2 FULL chains
```

API:

```sql
SELECT dbbackup.set_retention_policy(
  backup_set := 'prod_core',
  keep_logs_for := interval '14 days',
  keep_fulls_for := interval '31 days',
  keep_min_full_chains := 2,
  delete_from_storage := true
);

SELECT * FROM dbbackup.pg_dbbackup_retention_plan('prod_core');
SELECT dbbackup.pg_dbbackup_apply_retention('prod_core', dry_run := false);
```

Deletion rules:

- never delete a FULL while retained LOG/DIFF backups depend on it
- never delete the newest `keep_min_full_chains`
- delete using manifest-driven object lists
- mark local metadata deleted only after S3 deletion succeeds
- keep tombstone/audit records
- support dry-run
- retention should be idempotent
- if external lifecycle already deleted an object, mark it missing and continue

Production profiles:

```text
Normal SaaS:
  LOG every 5-15 minutes
  FULL daily
  PITR window 14 days
  FULL history 31-90 days

Higher compliance:
  LOG every 1-5 minutes
  FULL daily or weekly
  PITR window 30-35 days
  monthly/yearly archived FULL with object lock

Dev/low-cost:
  LOG hourly
  FULL daily or weekly
  keep 7 days
```

```sql
SELECT dbbackup.create_s3_target(...);
SELECT dbbackup.create_backup_set('prod_core', 'prod_s3');

  'prod_core', 'log-10m', 'log', every := interval '10 minutes'
);
SELECT dbbackup.set_retention_policy('prod_core', interval '14 days', interval '31 days', 2);
```

Ad hoc backup:

```sql
SELECT dbbackup.pg_dbbackup_to_storage(
  dbname := 'app',
  type := 'full',
  storage_target := 'prod_s3'
);
```

Search:

```sql
SELECT * FROM dbbackup.search_backups(
  dbname := 'app',
  from_time := (now() - interval '7 days')::timestamptz,
  to_time := now()::timestamptz,
  storage_target := 'prod_s3'

```

Restore:

```sql
SELECT dbbackup.pg_dbrestore_at(
  dbname := 'app',
  target_db := 'app_restore',
  stop_at := '2026-05-12 03:42:10+00'::timestamptz,

```

## Manual Restore Drills

Restore drills are supported as explicit jobs, not automatic schedules.

Example:

```sql
SELECT dbbackup.restore_drill(
  dbname := 'app',
  storage_target := 'prod_s3',
  stop_at := now() - interval '5 minutes',
  target_db := 'drill_app_20260512'
);

SELECT * FROM dbbackup.restore_drill_plan(
  backup_set := 'prod_core',
  sample_size := 3
);
```

## Events and Webhooks

Native events are the source of truth.

Event table shape:

```text
dbbackup.events
  event_id
  severity
  event_type
  backup_set
  dbname
  chain_id
  job_id
  message
  payload_json
  created_at
```

Emit `NOTIFY dbbackup_events` for new events.

Optional webhooks:

```sql
SELECT dbbackup.create_webhook(
  name := 'ops',
  url := 'https://example.com/dbbackup-events',
  secret_ref := 'env:DBBACKUP_WEBHOOK_SECRET',
  event_types := ARRAY['backup_failed', 'chain_broken', 'unsafe_failover']
);
```

Webhook delivery is async and retryable. Webhook failure records a delivery
event but must not fail backup, restore, or retention correctness.

## Remaining Implementation Risks

These are not open product decisions, but they need careful engineering.

1. Packaging native S3 dependencies on Linux and Windows.
2. Correct SigV4 behavior across AWS, MinIO, R2, and other compatible targets.
3. Streaming AES-GCM frame design, including random failure recovery and restore
   verification.
4. Background worker crash/restart behavior for partially completed jobs.
5. Multipart upload cleanup after postmaster crash.
6. Object Lock provider differences.
7. Webhook security and secret handling through environment references.
8. Bandwidth throttling without blocking PostgreSQL responsiveness.
9. Efficient manifest scanning for very large buckets.
10. Retention interactions with external S3 lifecycle policies.
11. Final SQL function names and signatures before v1 API freeze.

Function names in this document are proposed. Behavior is fixed by this plan,
but names/signatures can change during implementation before release.

## Definition of Done

S3/scheduler support is complete when:

1. FULL, DIFF, and LOG can upload to S3-compatible storage with multipart and
   checksum verification.
2. Restore can read artifacts from S3 and restore by explicit URI list.
3. `pg_dbrestore_at` can auto-select a continuous chain by timestamp.
4. Backup search returns only verified, continuous PITR ranges.
5. Backup sets and mutable database membership are implemented.
6. Scheduler supports named schedules, alter/pause/resume/drop, cron,
   interval, time zones, and wrap-midnight windows.
7. FULL schedules take a tail LOG before starting a new FULL chain.
8. Primary-change detection automatically takes LOG or reseed FULL per database.
9. Retention dry-run and apply are chain-safe and manifest-driven.
10. Search/restore automatically lazy-imports S3 manifests after cluster loss.
11. S3 Object Lock is validated and applied when configured.
12. Manual restore-drill jobs are available.
13. Event table, `NOTIFY`, and optional webhook delivery are implemented.
14. Tests cover AWS-compatible local S3, likely MinIO, plus failure cases.
15. Documentation covers environment credentials, IAM permissions for future
    support, retention, HA behavior, and restore drills.
